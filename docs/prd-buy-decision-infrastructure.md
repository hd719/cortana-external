# PRD: Buy Decision Infrastructure (Production-Safe)

**Status:** Proposed  
**Owner:** Cortana / OpenClaw trading workflow  
**Scope:** `cortana-external` backtester data and decision-support layers (consumed by existing `cortana` cron flow)  
**Intent:** Improve buy decision quality through better data, reusable features, and measurable calibration without adding fragility to the live path.

---

## Problem Statement

The system is operationally stronger after cron decoupling, but buy decisions still rely on infrastructure that is only partially standardized:

1. Data normalization is spread across modules, which makes feature behavior harder to compare across scan, quick-check, and research paths.
2. The live scan cap and ranking inputs are better than pure ordering, but the feature pipeline is not yet treated as one reusable contract.
3. Confidence and watchlist prioritization are not fully tied to a stable, walk-forward calibrated mapping.
4. Slippage and execution costs are discussed in overlays, but are not yet a consistent part of buy-decision evaluation gates.
5. Promotion of new ranking ideas still risks "looks smart" drift unless evidence thresholds are explicit and auditable.

Result: the stack can produce good signals, but decision quality is not yet maximally consistent, explainable, or measurable across time.

---

## Goals

1. Create one production-safe market-data and feature contract used by universe selection, quick-check, and research evaluation.
2. Improve top-120 scan quality with deterministic, precomputed ranking inputs.
3. Add confidence calibration that is explicit, measurable, and regime-aware.
4. Enforce walk-forward and cost-aware evaluation before promoting new buy-decision modifiers.
5. Improve operator clarity with concise output that explains "why this is a BUY/WATCH now" in practical terms.
6. Preserve current reliability model: base compute remains minimal; enrichments stay optional/fail-open.

## Non-Goals

- No wallet, broker, auth, or auto-order behavior.
- No replacement of the Python regime/technical engine as final authority.
- No heavy daytime network fanout in the blocking cron path.
- No unbounded online learning loop directly mutating live weights intraday.
- No "Polymarket-only strategy" path.

---

## Architecture Principles

1. **Python regime/technical engine stays primary.**
2. **Precompute first, consume fast.** Heavy work runs nightly or premarket, not inline during market-session compute.
3. **Deterministic fallback always exists.** Missing/stale optional artifacts must not fail the base path.
4. **Promotion requires evidence.** Anything that touches ranking/confidence must pass explicit walk-forward and cost-aware gates.
5. **Operator trust over model complexity.** Outputs should be interpretable in compact mode and richer in verbose mode.

---

## Proposed Architecture

### A) Normalized Market-Data Snapshot Layer (Precompute)

Build a single normalization pass for core tradable universe inputs:
- canonical OHLCV schema
- timezone-safe timestamps and session alignment
- consistent return/resampling behavior across all consumers
- artifact freshness metadata

Proposed artifacts:
- `backtester/data/cache/normalized_market_snapshot.json` (metadata + symbol index)
- `backtester/data/cache/normalized_features_latest.parquet` (or csv fallback)
- `backtester/data/cache/normalized_features_meta.json`

Rules:
- generated by nightly/pre-market workflow
- read-only in market session path
- if missing/stale, live path falls back to current deterministic behavior

### B) Reusable Feature Contract

Define a bounded feature set with stable names and units so ranking, quick-check, and outcome analysis use the same inputs.

Core feature families:
- trend/relative strength
- momentum across short/medium horizons
- volatility and downside stress
- liquidity/execution quality proxies
- pullback and distance-from-high structure
- regime alignment flags

Feature contract includes:
- schema version
- each feature definition
- valid ranges and null-handling rules
- freshness and provenance markers

### C) Ranked Universe Input Bridge

Extend current prefilter/ranking path so live scans consume the feature snapshot contract directly:
- reserve explicit priority symbols first
- score remaining symbols from precomputed features
- fill remaining scan slots deterministically
- keep strict fallback to deterministic ordering when cache is stale/unavailable

This keeps scan quality higher without reintroducing daytime fetch risk.

### D) Confidence Calibration Layer

Add a calibration component that maps scored setups to expected reliability by relevant slices:
- strategy family
- regime bucket
- stress/liquidity bucket

Outputs:
- calibration table artifact with version and timestamp
- reliability bins used for operator-facing confidence context
- stale-calibration guardrails (annotate as uncalibrated when stale)

Calibration is advisory and bounded. It does not bypass hard risk gates.

### E) Walk-Forward + Cost-Aware Evaluation Layer

Promotion decisions use forward outcomes with explicit assumptions:
- walk-forward windows (rolling train/eval)
- horizon metrics (`1d/5d/10d` as available)
- slippage/cost assumptions by liquidity tier
- downside and churn penalties

Any proposed ranking modifier must show positive net lift after costs and hold stability across windows before promotion.

### F) Promotion Gate and Registry

Use a file-backed registry for decision modifiers:
- stage: `research_only` -> `surfaced` -> `bounded_rank_modifier`
- thresholds: minimum samples, stability, cost-adjusted lift, downside controls
- demotion triggers and cooldown

Only bounded, allowlisted modifiers can influence ranking.

### G) Operator-Facing Outputs

Compact mode should answer:
- what is actionable now
- why top names surfaced (top factors only)
- whether confidence is calibrated or stale
- whether cost/liquidity warnings downgrade urgency

Verbose/JSON mode should include:
- top contributing features per name
- calibration bucket context
- estimated execution-quality tier
- promotion-state annotations where applicable

## Data and Feature Scope (V1)

Minimum reusable features to standardize first:
- returns: `1d`, `5d`, `20d`
- momentum vs baseline index
- trend quality (MA stack/slope family)
- realized volatility (`10d` and `20d`)
- drawdown from local high
- pullback quality and recovery shape
- dollar volume and liquidity tier
- adverse-regime alignment flag

Optional v1.5 additions:
- sector-relative momentum
- event proximity flags (earnings window)
- spread/impact proxy refinement

## Validation Plan

### Unit Tests

- normalization correctness and timezone/session handling
- feature computation determinism and null handling
- ranking score determinism with stable tie-breaking
- stale/missing cache fallback behavior
- calibration table generation and stale guards
- promotion-gate pass/fail/demotion logic

### Integration Tests

- nightly precompute writes valid feature artifacts
- live scan consumes feature artifacts without inline refresh
- quick-check and batch quick-check read shared feature/calibration artifacts
- alert formatting surfaces calibrated/stale status correctly

### Evaluation Tests

- walk-forward report generation with cost assumptions
- cost-adjusted lift checks by regime and liquidity buckets
- regression checks for cron runtime and failure rate stability

---

## Rollout Phases

### Phase 0: Baseline and Contract Freeze
- freeze baseline metrics for hit rate, forward return buckets, scan runtime, and notification reliability
- finalize feature schema v1 and artifact contract

### Phase 1: Normalized Data + Feature Snapshot
- implement precompute snapshot pipeline
- wire deterministic fallback behavior
- ship with no ranking-weight changes yet (shadow outputs allowed)

### Phase 2: Ranking Input Upgrade (Bounded)
- switch prefilter scoring to feature snapshot source
- keep existing priority-first behavior
- measure selection lift without changing decision authority

### Phase 3: Calibration + Cost-Aware Scoring
- generate calibration artifacts from settled outcomes
- add cost/slippage-aware evaluation metrics
- surface calibration/cost context in operator outputs

### Phase 4: Promotion Governance
- enable gated promotion into bounded rank modifiers
- enforce automatic demotion and cooldown on degradation
- keep all modifier influence capped and auditable

---

## Operator Workflow

1. Nightly/pre-market precompute refreshes normalized snapshot and feature artifacts.
2. Market-session base compute runs as today (regime + CANSLIM/Dip Buyer + guardrails).
3. Live ranking reads fresh feature artifacts; if stale, deterministic fallback stays active.
4. Quick-check/re-check reads the same feature and calibration contracts for consistent verdict context.
5. Notify surfaces compact actionable output and marks calibration/cost context clearly.
6. Weekly review checks walk-forward and cost-adjusted reports before any promotion decisions.

---

## Success Metrics

Primary:
- improved forward outcome quality for top-ranked scanned names vs baseline
- improved BUY/WATCH precision after cost assumptions
- no increase in market-session cron failure rate
- no regression in notify correctness/idempotency

Secondary:
- faster operator decision time due to clearer "why surfaced" context
- lower churn in promoted watchlists
- stable behavior across different regime buckets, not just one period

---

## Risks and Mitigations

- **Risk:** Overfitting from too many features  
  **Mitigation:** bounded feature set v1, walk-forward gates, sample minimums.

- **Risk:** Daytime path slowdown  
  **Mitigation:** precompute-first design, strict no-inline-refresh policy for live path.

- **Risk:** False confidence from stale calibration  
  **Mitigation:** freshness checks + explicit stale annotation + fallback behavior.

- **Risk:** Complexity creep in ranking modifiers  
  **Mitigation:** stage registry, bounded caps, manual approval for promotion.

- **Risk:** Decision authority drift away from base engine  
  **Mitigation:** hard rule that regime/technical engine remains final gate.
